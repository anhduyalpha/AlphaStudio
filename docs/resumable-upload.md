# Resumable upload architecture

AlphaStudio 3.6 adds a durable, streaming path for large files without removing
the existing `POST /api/uploads` multipart contract. The Converter chooses the
session path at 8 MiB; small files and older clients continue to use multipart.

## Protocol

1. `POST /api/upload-sessions/init` accepts `workspaceId`, `originalName`,
   declared `size`, optional `mime`, and optional `chunkSize`.
2. `PUT /api/upload-sessions/:id/chunks/:index` requires
   `Content-Type: application/octet-stream`, exact
   `Content-Range: bytes START-END/TOTAL`, and `X-Chunk-SHA256`.
3. `GET /api/upload-sessions/:id` returns committed indexes, actual received
   bytes, the next missing index, status, expiry, and finalized file id. It never
   exposes a local path.
4. `POST .../pause` and `POST .../resume` persist lifecycle state. A chunk that
   finishes while pause races it may be committed, and the returned byte count
   remains authoritative.
5. `POST .../finalize` rejects missing chunks, assembles them sequentially,
   performs bounded header/MIME/magic validation, and creates the file record.
6. `DELETE /api/upload-sessions/:id` cancels non-finalized sessions and removes
   only the UUID-confined staging directory.

Identical repeats of a committed index succeed only when range, size, and
SHA-256 match. A different body at the same index returns `409
UPLOAD_CONFLICT`. Short/long bodies, bad ranges, invalid checksums, unsupported
content, and configured size limits fail explicitly.

## Persistence and recovery

`upload_sessions` and `upload_chunks` are migration-v6 SQLite tables. Chunk
paths are generated by the server under `data/temp/upload-sessions/<uuid>/`;
client filenames never participate in a staging path. On server restart, a
session interrupted during finalize returns to `uploading`. Finalize also heals
the narrow commit window where a `files.upload_session_id` exists but session
completion was not recorded.

The browser stores only an opaque mapping from file metadata to session id.
Raw file access cannot survive reload by browser design, so the UI pauses the
persisted server session and asks the user to reselect the same file. The
controller then queries committed indexes and sends only missing chunks.

Expired sessions are deleted by the existing 15-minute maintenance loop.
Orphan directory cleanup accepts UUID-shaped children only, checks age, and
uses path confinement before recursive deletion. Completed sessions retain a
short metadata window but their chunks are removed immediately.

## Detection and duplicates

Finalize hands the assembled file to the existing workspace upload service.
Quick validation and detection read at most the configured header window
(currently 4,100 bytes); a size/head/tail fingerprint may reuse provisional
detection metadata. The file is inserted as `processing` and emitted over SSE
immediately. Full streaming SHA-256 and deep detection run asynchronously, then
emit terminal `ready` or `failed`, so the UI cannot remain indefinitely at
Inspecting.

Fingerprint equality is never identity. `duplicateOf` is written only after
full SHA-256 and size equal a prior ready file, preventing false duplicates for
files with identical heads/tails but different middle bytes.

## Limits and operations

| Variable | Default | Purpose |
|---|---:|---|
| `MAX_UPLOAD_BYTES` | 100 MiB | Maximum declared and assembled file size |
| `UPLOAD_CHUNK_BYTES` | 5 MiB | Server-selected chunk size, clamped to 256 KiB–16 MiB |
| `UPLOAD_CHUNK_TIMEOUT_MS` | 2 minutes | Stops a stalled request stream |
| `UPLOAD_SESSION_TTL_MS` | 24 hours | Inactive session/chunk retention |

The frontend limits uploads to three files at once and sends one chunk per file
at a time. Each client buffers only one bounded chunk for Web Crypto SHA-256;
the server streams request bodies and final assembly without a shell or a
whole-file buffer.
